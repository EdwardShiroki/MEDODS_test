# Task Service

Сервис для управления задачами с HTTP API на Go.

## Требования

- Go `1.23+`
- Docker и Docker Compose

## Быстрый запуск через Docker Compose

```bash
docker compose up --build
```

После запуска сервис будет доступен по адресу `http://localhost:8080`.

Если `postgres` уже запускался ранее со старой схемой, пересоздай volume:

```bash
docker compose down -v
docker compose up --build
```

Причина в том, что SQL-файл из `migrations/0001_create_tasks.up.sql` монтируется в `docker-entrypoint-initdb.d` и применяется только при инициализации пустого data volume.

- `PUT /api/v1/tasks/{id}`
- `DELETE /api/v1/tasks/{id}`

## Фича: Периодичность задач

Добавлена возможность создавать повторяющиеся задачи с различными типами периодичности. При создании задачи с настройками периодичности автоматически генерируются инстансы задач на будущие даты.

### Типы периодичности

- **daily**: Задачи создаются каждый N-й день (например, каждый 2-й день).
- **monthly**: Задачи создаются ежемесячно в указанное число месяца (1-30).
- **specific_date**: Задачи создаются только на указанные конкретные даты.
- **even_days**: Задачи создаются только в четные числа месяца.
- **odd_days**: Задачи создаются только в нечетные числа месяца.

### Поля для периодичности

- `recurrence_type` (string): Тип периодичности (опционально).
- `recurrence_value` (int): Значение для daily (N дней) или monthly (число месяца).
- `specific_dates` ([]time.Time): Массив дат для specific_date.
- `end_date` (time.Time): Дата окончания генерации (опционально, по умолчанию +1 месяц).

### Примеры использования

#### Создание ежедневной задачи (каждые 3 дня)
```json
POST /api/v1/tasks
{
  "title": "Ежедневный осмотр",
  "description": "Осматривать пациентов",
  "status": "new",
  "recurrence_type": "daily",
  "recurrence_value": 3,
  "end_date": "2026-05-01T00:00:00Z"
}
```

#### Создание ежемесячной задачи (15-го числа)
```json
POST /api/v1/tasks
{
  "title": "Месячный отчет",
  "description": "Подготовить отчет",
  "status": "new",
  "recurrence_type": "monthly",
  "recurrence_value": 15
}
```

#### Создание задач на конкретные даты
```json
POST /api/v1/tasks
{
  "title": "Встреча с поставщиками",
  "description": "Обсудить поставки",
  "status": "new",
  "recurrence_type": "specific_date",
  "specific_dates": ["2026-04-15T10:00:00Z", "2026-05-20T10:00:00Z"]
}
```

### Предположения и ограничения

- Генерация задач происходит только при создании (не при обновлении, чтобы избежать дубликатов).
- Если `end_date` не указана, задачи генерируются на 1 месяц вперед.
- Для `monthly`: значение 1-30; если в месяце меньше дней, задача не создается.
- Для `specific_date`: даты должны быть в будущем относительно времени создания.
- Периодичность не влияет на обновление/удаление существующих задач — только на создание новых инстансов.
- Валидация: обязательные поля проверяются, даты валидируются на корректность.

### Архитектура реализации

- **Domain**: Модель `Task` расширена полями периодичности.
- **Use Case**: Логика генерации задач в `generateRecurringTasks`.
- **Repository**: SQL-запросы обновлены для новых полей.
- **Transport**: DTO и handlers передают поля в use case.
- **БД**: Миграция добавляет колонки в таблицу `tasks`.
